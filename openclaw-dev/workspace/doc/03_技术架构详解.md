## 一、整体架构概览

### 1.1 架构设计原则

| 原则 | 说明 |
|-----|------|
| **云原生优先** | 基于 Kubernetes 构建，充分利用容器化优势 |
| **多租户隔离** | 网络、存储、计算三层隔离，保障租户安全 |
| **弹性伸缩** | 根据负载自动扩缩容，降低成本 |
| **可观测性** | 全链路监控、日志、告警 |
| **安全合规** | 等保 2.0 三级标准设计 |

## 二、系统分层设计

### 2.1 用户接入层

**功能：** 提供多种用户交互方式

| 组件 | 技术栈 | 说明 |

| Web 控制台 | React + Ant Design | 主控制台，支持实例管理、监控、计费 |

**功能：** 统一入口、安全防护、流量管理

| 组件 | 技术选型 | 功能 |

## 三、核心模块详解

### 3.1 GPU 调度器（自研核心）

**架构图：**
```
┌─────────────────────────────────────────────────────────┐
│                    GPU 调度器                            │
├─────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  拓扑感知模块  │  │  配额管理模块  │  │  抢占调度模块  │  │
│  │  - NVLink 优先 │  │  - 租户配额   │  │  - 优先级队列  │  │
│  │  - NUMA 感知   │  │  - 资源预留   │  │  - 抢占策略   │  │
│  │  - PCIe 拓扑   │  │  - 超额售卖   │  │  - 迁移机制   │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  MIG 切分模块  │  │  监控采集模块  │  │  故障处理模块  │  │
│  │  - A100 切分   │  │  - 使用率     │  │  - 健康检查   │  │
│  │  - 实例管理   │  │  - 显存占用   │  │  - 自动迁移   │  │
│  │  - 资源映射   │  │  - 功率监控   │  │  - 告警通知   │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
└─────────────────────────────────────────────────────────┘
```

**调度流程：**
```
1. 用户提交请求（GPU 型号、数量、时长）
         │
         ▼
2. 检查租户配额（是否超限）
         │
         ▼
3. 查询可用资源（拓扑感知）
         │
    ┌────┴────┐
    │         │
    ▼         ▼
4a. 有资源    4b. 无资源
    │         │
    ▼         ▼
5. 创建 Pod   5. 加入等待队列
    │         │
    ▼         ▼
6. 挂载存储   6. 资源释放后调度
    │
    ▼
7. 启动实例
```

**调度策略：**
| 策略 | 说明 | 优先级 |
|-----|------|--------|
| NVLink 优先 | 同机 8 卡优先调度 | P0 |
| 拓扑感知 | 感知 NUMA、PCIe 拓扑 | P0 |
| 抢占调度 | 高优先级可抢占低优先级 | P1 |
| MIG 切分 | A100/H100 支持 7 实例切分 | P1 |
| 空闲回收 | 检测到空闲自动释放 | P2 |

### 3.2 计费服务

**架构设计：**
```
┌─────────────────────────────────────────────────────────┐
│                      计费服务                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐ │
│  │  计量采集   │───▶│  费用计算   │───▶│  账单生成   │ │
│  │  (10 秒/次)  │    │  (实时)     │    │  (每日)     │ │
│  └─────────────┘    └─────────────┘    └─────────────┘ │
│         │                  │                  │         │
│         ▼                  ▼                  ▼         │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐ │
│  │ Prometheus  │    │ 计费规则引擎 │    │  支付对接   │ │
│  │  Exporter   │    │ - 按时长     │    │ - 微信      │ │
│  │             │    │ - 按用量     │    │ - 支付宝    │ │
│  │             │    │ - 套餐折扣   │    │ - 对公转账  │ │
│  └─────────────┘    └─────────────┘    └─────────────┘ │
└─────────────────────────────────────────────────────────┘
```

**计费维度：**
| 维度 | 采集方式 | 精度 |
|-----|---------|------|
| GPU 时长 | K8s Pod 运行时间 | 秒级 |
| GPU 使用率 | DCGM Exporter | 10 秒 |
| 显存占用 | DCGM Exporter | 10 秒 |
| 存储用量 | Ceph 监控 | 分钟级 |
| 网络流量 | Traefik 日志 | 实时 |

**计费公式：**
```
费用 = Σ(资源单价 × 使用时长 × 折扣系数)

示例：
A100 80G: 15 元/小时
使用 2.5 小时，套餐 8 折
费用 = 15 × 2.5 × 0.8 = 30 元
```

### 6.2 分布式存储方案

**选型对比：**
| 方案 | 优势 | 劣势 | 适用场景 |
|-----|------|------|---------|
| Ceph | 功能全、生态好 | 运维复杂 | 通用存储 |
| GlusterFS | 简单、易扩展 | 小文件性能差 | 大文件存储 |
| NFS | 简单、兼容好 | 单点瓶颈 | 小规模 |
| Lustre | 高性能 | 成本高 | HPC 场景 |

**我们的选择：Ceph**
- RBD: 块存储（系统盘）
- CephFS: 文件存储（用户数据）
- RGW: 对象存储（备份归档）

### 6.3 数据持久化方案

```
用户实例 (Pod)
     │
     │ PVC 挂载
     ▼
CephFS (分布式文件系统)
     │
     │ 3 副本
     ▼
OSD 节点 (物理磁盘)
     │
     │ 每日快照
     ▼
备份存储 (异地)
```

**数据保护：**
- 存储 3 副本，允许 2 节点故障
- 每日自动快照，保留 7 天
- 异地备份，防止机房级故障

### 8.2 故障转移策略

| 故障类型 | 检测方式 | 恢复策略 | RTO |
|---------|---------|---------|-----|
| Pod 故障 | K8s 健康检查 | 自动重启 | <30 秒 |
| 节点故障 | Node 心跳 | Pod 迁移 | <2 分钟 |
| Master 故障 | 心跳检测 | VIP 漂移 | <10 秒 |
| 存储故障 | I/O 超时 | 副本切换 | <30 秒 |
| 网络故障 | 连通性检测 | 路由切换 | <10 秒 |
